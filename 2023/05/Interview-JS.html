eval

eval 是 JS 全局函数，接受一个字符串参数，并将字符串转换成 Js 代码来执行。
eval 函数的作用是动态执行 JavaScript 代码，将字符串转换成可执行的 js 代码。

eval 函数缺点

  1. 存在安全隐患
    虽然 eval 可以将字符串转成可执行的 js 代码，但存在安全隐患。
    如果使用 eval 函数执行的字符串来自用户输入，那将会导致安全问题，所以我们应尽量避免使用 eval 函数，尤其在处理用户输入的数据或者不可信来源数据时。如果需要使用 eval 函数，那么要确保 eval 函数输入的字符参数来自可信来源。

所以我们在使用 eval 函数时，要确保参数的验证和过滤。

  1. 性能消耗

eval 函数是动态执行的，执行 eval 函数需要两步，一：解析成 JS 代码，二：执行 JS 代码。这会使代码执行速度变慢，并消耗性能

请问 Vue 和 React 的区别是什么？

https://juejin.cn/post/6844904078196097031#heading-65

  1. 核心思想和写法差异

Vue
+ vue 推崇灵活易用（渐进式开发框架），数据可变，双向数据绑定（依赖收集）
+ vue 推崇template 、单文件 vue

React
+ react 推崇函数式编程（纯函数组件），数据不可变，单向数据流
+ react 推崇使用 JSX HOC，all in js

  1. 响应式原理不同
    Vue

    • Vue 依赖收集，自动优化，数据可变
    • Vue 递归监听 data 的所有属性，直接修改
    • 当数据改变时，自动找到引用组件重新渲染

    React

    • React 基于状态机制，手动优化，数据不可变，需要setState驱动新的state 替换老的state。
    • 当数据改变时，以组件为根目标，默认全部重新渲染。

  2. diff 算法不同
    Vue 和 React diff 在流程思维上是类似的，都是基于两个假设（使得算法复杂度降为O(n)）

    • 不同组件产生的不同DOM结构。当type 不相同时，对应 DOM 操作就是直接销毁老的DOM，创建新的 DOM。

    • 同一层次的一组子节点，可以通过唯一的 key 区分。

      两者在源码实现上有区别：

    Vue

    • Vue 基于snabbDom 库，它有较好的速度以及模块机制。+ Vue diff 使用双端对比，边对比，边更新DOM

    React

    • React 主要使用 diff 队列保存需要更新的 DOM，得到patch树，再统一操作批量更新 DOM

  3. 事件机制不同
    Vue

    • vue 原生事件使用标准 web 事件
    • vue 组件自定义事件机制，是父子组件通信的基础
    • vue 合理利用了 snabbdom 库的模块插件

    React

    • react 原生事件被包装，所有事件都冒泡到顶层 document 监听，然后在这里合成事件下发事件机制，而不是和 web dom 强绑定
    • react 组件上无事件，父子组件通信使用 props

React 的单向数据流和 Vue 的双向数据绑定的区别

vue会在初始化时收集所有的依赖，数据改变view跟着改变
react的话改变数据需要手动调用setState方法

React 单向数据流的优缺点

优点： 所有变化都可以被追踪、记录，状态变化通过手动调用更新，容易溯源

  1. 可预测性
  2. 组件解耦
  3. 性能优化： 通过单向数据流，react 可以更精确地追踪数据变化，并做出局部渲染的优化。react 使用虚拟 DOM 进行渲染，通过比较前后状态的差异，只更新变化的部分，减少不必要的渲染和重绘
    缺点：会有很多类似的样板代码，代码量上升

说下幽灵依赖，为什么会有幽灵依赖，pnpm除了下载速度快还有什么好处

幽灵依赖指的是：组件包A依赖组件包B，B依赖C，那么在组件包A中没有声明C这个依赖时，也可以直接使用C。这就是幽灵依赖
幽灵依赖是在使用 npm/yarn 进行包管理时，由于 node_module 的扁平结构引起的。

在 pnpm 中，它不允许安装 package.json 中不存在的包，并且使用软链接很好的解决了幽灵依赖的问题。pnpm 使用软链接来创建依赖项的嵌套结构，将项目的直接依赖链接到 node_modules 的根目录，直接依赖的实际位置在 .pnpm/@/node_modules, 依赖包中的每个文件再硬链接到 .pnpm store

在用Vite的时候，开发环境和生产环境的不一致是什么原因导致的，怎么解决

vite 是一个基于浏览器原生EsModlues 的开发服务器，利用浏览器去解析模块，在服务端按需编译返回。完全跳过了打包的这个阶段
vite 的主要优势是在开发阶段。DEV 是用 Esbuild对代码进行构建，但是 PROD 使用的是 rollup。
Esbuild 在以下几个场景难以处理：

  • 自动 css 代码分割
  • 对异步懒加载请求自动优化
  • 手动的代码分割控制

由于 PROD 的构建场景和 DEV 开发阶段存在天然差异，所以不可避免的生产环境的表现会跟开发环境不一致。
要解决这个问题，需要保证生产环境依赖处理规则和开发环境一致，使用 ESBuild 进行打包，目前可以通过添加 optimizeDeps.disabled: false 配置开启。

宏任务和微任务

宏任务和微任务是 JavaScript 中用来管理异步操作的两种任务类型。它们在执行顺序上有一定的差别。

宏任务：会在同步任务执行完成后执行。
微任务：会在当前宏任务执行完成后立即执行。

事件循环（event loop）

由于 Js 是单线程运行的，所以在代码执行时会根据当前上下文时，通过将不同函数压入执行栈中来保证代码的有序运行。当代码执行时，同步任务未执行完毕时遇到异步任务，会将这个异步任务挂起，执行完同步任务后在来执行异步任务。

异步任务有两种任务类型，分别为宏任务和微任务。

  • 宏任务：会在同步任务执行完成后执行。
  • 微任务：会在当前宏任务执行完成后立即执行。

浏览器的事件循环

在浏览器环境中，宏任务和微任务是通过事件循环机制来管理的，在每次事件循环中，浏览器会执行所有的宏任务，再执行所有的微任务。这个过程不断重复直到所有的宏任务和微任务执行完毕。

浏览器事件循环过程：
1. 从宏任务队列中出队，并执行最早的任务
2. 执行所有的微任务
a. 当微任务队列非空时，出队执行最早的微任务
3. 如果有变更，则将变更渲染出来
4. 如果宏任务队列为空，则休眠直到出现宏任务
5. 转到步骤1

Node的事件循环

在 Node.js 中也是通过事件循环机制来管理宏任务和微任务。但它与浏览器的事件循环机制有所不同，它是通过 libuv 库来完成的，在 Node.js 中，它会先执行所有的宏任务，再执行所有的微任务，最后在执行I/O操作的回调。这样的机制使用 Node.js 在处理 I/O 操作时有更好的性能。
